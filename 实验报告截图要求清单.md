# 实验报告提交与截图要求清单

这份文件是对课程文档中《实验报告提交及评分标准.md》的整理版。原文写得比较散，可以简单理解为：报告要证明你真的完成了环境配置、代码实现、测试运行、模型对比和 UI 演示。

## 一、报告整体结构

实验报告建议按下面顺序写：

1. 封面信息
2. 实验目的
3. 实验环境
4. 实验内容与步骤
5. 实验结果与分析
6. 理论问题回答
7. 实验总结与心得体会

封面需要写：

- 实验名称：传统 SaaS 到 AI Agent (MCP) 全景实战
- 学生姓名
- 学号
- 班级
- 实验日期
- 指导教师

- Dense 模型是 `qwen3.6:27b`。
- MoE 模型是 `qwen3.6:35b-a3b`。
- 如果使用 GGUF 量化模型，要写明量化版本和原因，例如 16GB MacBook Pro 需要低内存量化。

- Agent 调度器的核心流程是：思考、调用工具、观察结果、继续决策、最终回答。

### 任务5：Dense 与 MoE 模型对比实验

要证明你分别用两个模型跑过同一个 Agent 任务。

建议截图：

1. Gradio 右侧模型选择为 `qwen3.6:27b`，输入任务并完成运行的截图。
2. Gradio 右侧模型选择为 `qwen3.6:35b-a3b`，输入同样任务并完成运行的截图。
3. 可选：两次运行的日志或耗时截图。

推荐测试输入保持一致：

```text
帮我查一下全公司的工资，算好扣税，然后给我个 CSV 文件
```

报告里要分析：

- Dense 模型：所有参数都参与推理，表现相对稳定但资源占用大。
- MoE 模型：每次只激活部分专家，适合复杂任务和多工具调用，通常更省有效计算。
- 对比时可以写：哪个模型更容易完成工具链调用、哪个更快、哪个更容易出错。

## 三、理论问题要回答什么

理论题一共 20 分，四道题。不用截图，写文字回答即可。

### 问题1：SaaS 与 Agent 架构对比

需要回答：

- 传统 SaaS 是程序员预先写死流程。
- AI Agent 是模型理解用户意图，再动态选择工具。
- “人适应机器”指用户必须按系统菜单和固定流程操作。
- “机器适应人”指系统理解自然语言，并主动规划执行路径。

### 问题2：Dense 与 MoE 架构

需要回答：

- Dense 模型每次推理基本使用全部参数。
- MoE 模型由多个专家组成，每次只激活部分专家。
- 多工具链式调用需要规划、参数构造、结果理解，MoE 在复杂任务分工上更有优势。

### 问题3：MCP 风格工具定义与 Agent 调度

需要回答：

- MCP 是 Model Context Protocol，用于让模型以统一协议连接工具、资源和上下文。
- 本实验只做 MCP 风格 Schema，因为课堂重点是理解工具调用思想，不需要搭完整 MCP Server。
- Schema 关键字段包括 `name`、`description`、`parameters`、`required`。
- Agent 调度器通过“思考、行动、观察”循环完成任务。

### 问题4：AI 时代软件工程

需要回答：

- 原子化业务工具库是把业务能力拆成可独立调用的小工具。
- 原子化可以让 Agent 更容易组合工具，也方便测试和复用。
- AI 时代的软件工程从“写死流程”转向“提供工具、约束边界、让模型调度”。

1. 本次实验完成了什么：
   - Ollama 本地模型部署
   - SaaS 固定流程
   - Agent 工具调用流程
   - Dense 与 MoE 对比
   - Gradio 双轨 UI

2. 掌握了什么技能：
   - Python 业务函数封装
   - JSON Schema 工具定义
   - OpenAI 兼容 API 调用 Ollama
   - Gradio UI 搭建
   - 单元测试与端到端验证

3. 遇到的问题和解决方案：
   - 模型太大：使用低内存 GGUF 量化。
   - 端口占用：换 Gradio 端口。
   - Agent 参数错误：补全 Schema 的 `parameters` 和 `required`。
   - 本地依赖问题：使用 Conda 专用环境。
